## 스코프

식별자(변수, 함수 등)가 자신이 선언된 위치에 의해 다른 코드가 자신을 참조할 수 있는 유효 범위

### 스코프의 종류

| **구분** | **설명**              | **스코프**  | **변수**  |
| -------- | --------------------- | ----------- | --------- |
| 전역     | 코드의 가장 바깥 영역 | 전역 스코프 | 전역 변수 |
| 지역     | 함수 몸체 내부        | 지역 스코프 | 지역 변수 |

### 스코프 체인

- 스코프는 함수의 중첩에 의해 계층적 구조를 갖음
- 자바스크립트 엔진은 스코프 체인을 따라
  변수를 참조하는 코드의 스코프에서 시작해서
  **상위 스코프 방향으로 이동**하며 선언된 변수를 검색
- 스코프 체인은 **‘물리적 실체’**로 존재한다.
  - `코드 실행 전`
    자바스크립트 엔진은 코드를 실행하기 전  
     **메모리의 일정 영역을 차지**하는 (이미지와 유사한) 자료구조(렉시컬 환경) 생성
  - `변수 선언 실행 시`
    변수 식별자가 해당 자료구조에 key로 등록
  - `변수 할당 시`
    자료구조의 변수 식별자에 해당하는 값 변경

### 함수 레벨 / 블록 레벨 스코프

- **블록 레벨 스코프**
  - 대부분의 프로그래밍 언어에서 채택
  - 함수 몸체뿐만 아니라 모든 코드 블록(if, for, while..)은 지역 스코프를 만듦
- **함수 레벨 스코프**
  - 오직 함수의 코드 블록(함수 몸체)만을 지역 스코프로 인정
  - `var` 키워드로 선언된 변수가 이에 해당

### 렉시컬 스코프

- **함수를 어디서 정의**했는지에 따라 함수의 상위 스코프를 결정
  → ‘함수를 어디서 호출했는지’가 아님!
- 함수의 상위 스코프는 언제나 자신이 정의된 스코프임
- 함수의 상위 스코프는 함수 정의가 실행될 때 정적으로 결정
  → 함수 정의가 실행되어 생성된 함수 객체는 이렇게 결정된 상위 스코프를 기억함
  (함수가 호출될 때마다 참조하기 위해서!)
  **“모든 함수는 자신의 상위 스코프를 기억한다”**

```tsx
var x = 1;

function foo() {
  var x = 10;
  bar();
}

function bar() {
  console.log(x);
}

foo(); // 1 -> 자신이 정의된 전역 스코프를 상위 스코프로 사용!
bar(); // 1
```

<aside>
💡 렉시컬 환경의 “외부 렉시컬 환경에 대한 참조”에 저장할 참조값, 즉 상위 스코프에 대한 참조는 함수 정의가 평가되는 시점에 함수가 정의된 환경(위치)에 의해 결정된다. 이것이 바로 렉시컬 스코프다.

</aside>

### 변수의 생명 주기

**지역 변수**

- 함수 내부에서 선언된 지역 변수의 생명 주기는 대부분 함수의 생명 주기와 일치함.
  (함수 종료 → 스코프 소멸) 1. 함수 내부에서 선언된 지역 변수는 함수가 생성한 스코프(렉시컬 환경)에 등록 2. 이 스코프가 소멸(메모리 해제)될 때 함께 소멸됨.
- 하지만, 지역 변수가 함수보다 오래 생존하는 경우가 있음.
  → 함수는 종료되었지만, 다른 코드에서 해당 스코프를 여전히 참조하고 있다면 스코프는 생존!

**전역 변수**

- ES6 모듈 사용 시 전역 변수 사용 불가
- ES6 모듈은 파일 자체의 독자적인 모듈 스코프 제공
  → 모듈 내에서 `var` 키워드로 선언한 변수는 더는 전역 변수가 아니며 window 객체의 프로퍼티도 아님!

### 변수 호이스팅

**`var`**

런타임 이전에 ‘선언 단계’와 ‘초기화 단계’가 한번에 진행됨

```tsx
// 1. 이 시점(런타임 이전)에는 변수 호이스팅에 의해 **선언**
// 2. undefined로 **초기화**
console.log(foo); // undefined

// 3. 변수에 값 할당
foo = 123;

console.log(foo); // 123

var foo;
```

**`const` / `let`**

런타임 이전에 ‘선언 단계’와 ‘초기화 단계’가 분리되어 진행됨

```tsx
// 1. 이 시점(런타임 이전)에는 변수 호이스팅에 의해 **선언**, 초기화x
// 초기화 이전의 일시적 사각지대(TDZ)에서 변수 참조 불가
console.log(foo); // ReferenceError: foo is not defined

// 2. 변수 선언문에서 **초기화**
let foo; // <- TDZ 종료
console.log(foo); // undefined

// 3. 변수 할당
foo = 1;
var foo; // 1
```

### **호이스팅 동작 과정**

1. 실행 컨텍스트 평가 단계

   - 자바스크립트 엔진은 코드를 한 줄씩 실행(런타임)하기 전에,
     전체 코드를 먼저 스캔하여 식별자들을 수집
   - `var`, `let`, `const`로 선언된 변수들 모두 렉시컬 환경에 메모리 공간을 확보하고 이름 등록

   ⇒ "런타임 이전에 변수가 렉시컬 환경에 수집 및 등록되는 현상"을 호이스팅이라고 부름

2. 바인딩 단계
   - `var`
     평가 단계에서 렉시컬 환경에 식별자가 등록됨과 동시에,
     메모리에 `undefined`를 할당하여 초기화까지 완료
     → 변수 접근 시 `undefined` 반환
   - `let` / `const`
     평가 단계에서 렉시컬 환경에 식별자는 등록되지만,
     메모리에 어떠한 값도 할당하지 않은(초기화되지 않은) 상태로 둠.
     → 이 상태의 변수에 접근 시 `ReferenceError` 발생

> 자바스크립트는 ES6에서 도입된 let, const를 포함해서 모든 선언을 호이스팅한다.
> 단, ES6에서 도입된 let, const, class를 사용한 선언문은 호이스팅이 발생하지 않는 것처럼 동작한다.

**“호이스팅이 발생하지 않는 것처럼 동작한다”의 의미?**

→ 식별자가 메모리(렉시컬 환경)에 미리 등록은 되지만(호이스팅 됨),
실제 코드 실행이 변수 선언에 도달하기 전까지 접근을 엄격하게 차단(TDZ)하기 때문에
마치 호이스팅이 일어나지 않은 것처럼 에러를 뱉는다

<aside>

**TDZ**

- 스코프의 시작 지점부터 초기화 시작 지점까지 변수를 참조할 수 없는 구간
- 초기화 단계가 실행되기 이전에 변수에 접근하려고 하면 참조 에러가 발생
- `let`, `const` 키워드로 선언한 변수는 스코프의 시작 지점부터 초기화 단계 시작 지점(변수 선언문)까지 변수 참조 불가
</aside>

<aside>

**초기화 vs 할당**

- 초기화
  - 자바스크립트 엔진이 변수를 다루기 위해 메모리 공간을 실제로 확보하고,
    최초의 기본값을 채워 넣어 변수를 참조할 수 있는 상태로 만드는 것
  - 엔진은 초기화 단계에서 기본값으로 `undefined` 를 넣음
- 할당 - 런타임에 할당 연산자(`=`)를 통해 개발자가 의도한 특정한 실제 값을 변수에 저장 - 초기화를 통해 이미 확보된 메모리 공간의 값을 새로운 값으로 덮어쓰거나,
새로운 메모리 주소를 가리키도록 연결 변경
</aside>
