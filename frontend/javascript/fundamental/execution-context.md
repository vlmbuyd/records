## 소스코드의 평가와 실행

![image (12).png](<attachment:125e85d0-e481-4583-a190-162078204bb4:image_(12).png>)

자바스크립트 엔진은 소스코드를 2개의 과정,
“소스코드 평가” → “소스코드 실행” 과정으로 나누어 처리한다.

### 1. 소스코드 평가

- 실행 컨텍스트 생성
- 변수, 함수 등의 선언문만 먼저 실행
  → 생성된 변수, 함수 식별자를 key로 하여 실행 컨텍스트가 관리하는 스코프 (렉시컬 환경의 환경 레코드)에 등록

### 2. 소스코드 실행

- 선언문을 제외한 소스코드 순차적 실행 (런타임 시작)
- 소스코드 실행 시 필요한 변수, 함수의 참조를 렉시컬 환경에서 검색하여 취득
  → 변수 값 변경 등 소스코드의 실행 결과를 다시 렉시컬 환경에 업데이트

## 실행 컨텍스트

- 소스코드를 실행하는 데 필요한 환경을 제공하고 코드의 실행 결과를 실제로 관리하는 영역
- 식별자를 등록 및 관리하는 스코프 & 코드 실행 순서 관리를 구현한 내부 메커니즘
- 전역 코드가 평가되거나 함수가 호출될 때마다 하나씩 생성되어 콜 스택에 푸시되는 개별 객체

<aside>
💡

모든 코드는 실행 컨텍스트를 통해 실행되고 관리된다!

</aside>

구체적으로 다음의 항목들을 관리한다.

1. 선언에 의해 생성된 모든 **식별자**(변수, 함수 등)를 스코프를 구분하여 **등록**하고
   상태 변화(식별자에 바인딩된 값의 변화)를 **지속적으로 관리**
2. 중첩 관계에 의해 생성된 스코프 체인을 통해 (상위 스코프로 이동하며) **식별자 검색**
3. 현재 실행 중인 **코드의 실행 순서를 변경** 가능해야 한다.
   (e.g. 실행 도중 함수 호출 시 현재 실행 코드 일시 중단 후 실행 순서를 변경하여 함수 내부로 진입)
   현재 실행 중이었던 코드로 복귀할 수도 있어야 한다.

<aside>
💡

식별자와 스코프는 실행 컨텍스트의 **렉시컬 환경**으로
코드 실행 순서는 **실행 컨텍스트 스택**으로 관리

</aside>

## 렉시컬 환경

![image (13).png](<attachment:a6fa8cde-fcf4-4e37-af00-4d6611e62619:image_(13).png>)

- 식별자와 식별자에 바인딩된 값, 상위 스코프에 대한 참조를 기록하는 자료구조로
  실행 컨텍스트를 구성하는 컴포넌트 - ‘식별자’와 ‘스코프’를 관리! - 각 실행 컨텍스트마다 1개씩 생성
- 키와 값을 갖는 객체 형태의 스코프(전역, 함수, 블록 스코프)를 생성하여
  식별자를 키로 등록하고 식별자에 바인딩된 값 관리
      → 스코프를 구분하여 식별자를 등록 및 관리하는 ‘렉시컬 스코프’의 실체!
- 렉시컬 환경은 다음 두 개의 컴포넌트로 구성
  - **환경 레코드**
    스코프에 포함된 식별자를 등록하고 등록된 식별자에 바인딩된 값을 관리하는 저장소
  - **외부 렉시컬 환경에 대한 참조**
    - 상위 스코프 참조를 통해 스코프 체인을 구현함
    - 외부 렉시컬 환경에 대한 참조에 ‘함수 정의가 평가된 시점에 실행 중인 실행 컨텍스트의 렉시컬 환경의 참조’가 할당
      - 풀어서 해석
        “함수가 호출되어 렉시컬 환경이 만들어질 때 상위 스코프를 가리키는 포인터(외부 렉시컬 환경에 대한 참조)에는, 런타임의 호출 위치가 아니라 엔진이 코드를 처음 해석할 때 미리 기억해 두었던 정의 위치의 상위 스코프(평가 시점의 렉시컬 환경 참조)가 그대로 할당된다.”
    ```tsx
    const x = 10; // (A) 전역 렉시컬 환경 (상위 스코프)

    function foo() {
      // (B) 함수 정의가 평가되는 시점
      console.log(x);
    }

    foo(); // (C) 함수가 호출되어 '외부 렉시컬 환경에 대한 참조'가 필요한 시점
    ```
    1. **함수 정의 평가 시점:** 자바스크립트 엔진이 위에서부터 코드를 읽다가 `(B)` 줄에서 `foo` 함수 선언을 발견하고 함수 객체를 만듦.
    2. 실행 중인 실행 컨텍스트의 렉시컬 환경: 이때 엔진은 전역 코드를 평가하고 실행하는 중이므로, 현재 바탕이 되는 환경은 전역 스코프 `(A)`이다.
    3. 엔진은 나중을 위해 이 `(A)`의 주소값을 함수 객체 내부에 조용히 저장해 둠.
    4. 시간이 흘러 `(C)`에서 `foo()`가 호출됩니다. `foo` 함수만을 위한 새로운 렉시컬 환경이 만들어집니다.
    5. **외부 렉시컬 환경에 대한 참조**: `foo` 함수 안에서 `x`를 찾으려면 밖으로 나가야 하므로 바깥 스코프를 가리키는 연결선이 필요해진다.
    6. 할당: 이때 엔진은 3번 과정에서 미리 저장해 두었던 `(A)`의 주소값을 꺼내어, 방금 만든 5번의 연결선과 연결한다.

## 콜 스택 (실행 컨텍스트 스택)

- 자바스크립트 엔진(정확히는 메인 스레드) 상에 오직 **1개**만 생성
- 생성된 실행 컨텍스트들을 담아두고, 어떤 순서로 코드의 제어권을 넘길지 관리

<aside>
💡

콜 스택의 최상위에 존재하는 실행 컨텍스트는 언제나 현재 실행 중인 코드의 실행 컨텍스트다.

</aside>

<aside>
💡

실행 컨텍스트는 함수의 정의될 때가 아니라 **호출**될 때 생성된다.

- **함수 정의(평가) 시점**
  자바스크립트 엔진이 코드를 읽어 내려가다 함수 선언문을 만나면, 함수 객체를 생성.
  이때 실행 컨텍스트를 만들지는 않지만, 해당 함수가 어디서 정의되었는지 그 상위 스코프에 대한 참조를 함수 객체의 내부 슬롯인 `[[Environment]]`에 저장
- **함수 호출(실행) 시점**
  실제로 함수가 호출되면, 비로소 자바스크립트 엔진은 이 함수의 코드를 실행하기 위한 환경 구축. 이때 생성되는 것이 바로 **실행 컨텍스트**

</aside>

<aside>
❓

각각의 실행 컨텍스트마다 렉시컬 환경과 스택이 생기는가?

- 실행 컨텍스트 스택(콜스택)은 전체 자바스크립트 엔진 내에 **오직 1개만 존재**
  자바스크립트는 싱글 스레드 언어. 따라서 전체 흐름 제어를 담당하는 스택은 단 하나.
  (각각의 함수마다 스택이 생기는 것이 아님!)
- 렉시컬 환경은 각각의 실행 컨텍스트마다 1개씩 생성
</aside>

<aside>
❓

`[[ThisValue]]`

함수가 호출되어 ‘실행 컨텍스트’가 만들어질 때, 해당 함수에 대한 ‘환경 레코드’도 함께 생성.
이 함수 환경 레코드 내부에 있는 여러 슬롯 중 `this` 키워드가 현재 어떤 객체를 가리키고 있는지 기억해 두기 위한 전용 보관함

`[[ ]]`: 이중 대괄호로 감싸진 용어는 ‘자바스크립트 엔진이 내부적으로 상태를 추적하고 구동하기 위해 사용하는 숨겨진 메모리 공간(상태 저장소)임.
개발자가 코드 상에서 직접 접근할 수 없으며, 엔진이 명세서의 규칙대로 정확히 동작하기 위해 자기 자신만 읽고 쓰는 엔진 전용 변수

</aside>

### 콜스택 동작 과정

```tsx
const x = 1;

foo();

function bar() {...}
```

1. 프로그램이 시작되면 엔진은 단 하나의 **실행 컨텍스트 스택(콜스택)** 준비
2. 가장 먼저 전역 코드를 평가하여 **전역 실행 컨텍스트**를 생성하고, 콜스택에 푸시
3. 코드를 실행하다가 `foo()` 호출을 만남
4. 엔진은 `foo` 함수를 위한 새로운 **`foo` 실행 컨텍스트**를 생성
   - 함수 내부에 선언된 지역 변수, 매개 변수, 내부 함수 등의 식별자를 등록하고
     관리하기 위한 저장소인 **지역 렉시컬 환경 객체를 별도로 생성**
   - 방금 생성한 이 지역 렉시컬 환경을 `foo` 실행 컨텍스트에 바인딩
5. 생성된 `foo` 실행 컨텍스트를 **실행 컨텍스트 스택**의 최상단에 푸시
6. `foo` 함수의 실행이 모두 종료되면, `foo` 실행 컨텍스트는 **스택에서 pop되어 제거**
7. 다시 스택의 최상단에 남은 전역 실행 컨텍스트로 제어권이 돌아가 나머지 코드를 마저 실행
